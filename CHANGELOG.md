# Changelog

Все заметные изменения проекта фиксируются в этом файле.

Формат основан на Keep a Changelog,
а версияция следует Semantic Versioning.

## [0.5.43] - 2026-03-07

### Changed
- LLM-классификатор вопросов Group Knowledge теперь учитывает контекст медиа: подпись к изображению (`caption`) передаётся как сильный сигнал вопроса, а доступное `image_description` добавляется в метаданные prompt для более точной классификации коротких подписей к скриншотам ошибок.

## [0.5.42] - 2026-03-07

### Added
- В `scripts/gk_collector.py` добавлен режим `--fill-missing-is-question`, который заполняет question-классификацию для уже сохранённых сообщений с `is_question IS NULL` без запуска Telethon listener.
- В `scripts/gk_analyze.py` добавлен режим полного rebuild Q&A-пар: `--all-dates --rebuild-pairs` удаляет старые пары и их vector-точки, а затем заново анализирует всю историю сообщений выбранных групп.

### Changed
- В документации Group Knowledge добавлены отдельные инструкции для массового заполнения `is_question` и полного переанализа всех Q&A-пар после изменений логики thread-анализа.

## [0.5.41] - 2026-03-07

### Added
- В настройки Group Knowledge добавлены `GK_ANALYSIS_QUESTION_CONFIDENCE_THRESHOLD` и `GK_INCLUDE_LLM_INFERRED_ANSWERS`, чтобы отдельно управлять жёстким распознаванием вопросов в thread-анализе и использованием `llm_inferred` пар в автоответчике.

### Changed
- `qa_analyzer.py` теперь учитывает сохранённые `is_question` и `question_confidence` не только в `llm_inferred`, но и в thread-chain анализе: сообщение с confidence выше порога считается вопросом цепочки и передаётся в LLM-валидацию как основной вопрос.
- Thread-контекст для LLM-валидации теперь тоже содержит `QUESTION_HINT` / `NOT_QUESTION_HINT`, чтобы модель видела сохранённую классификацию всех сообщений цепочки.

### Fixed
- Поиск ответов Group Knowledge теперь умеет глобально исключать `llm_inferred` пары из BM25, vector и финального LLM-контекста, если это отключено через настройки.

## [0.5.40] - 2026-03-07

### Changed
- В логах Group Knowledge backfill теперь показывается прогресс по просмотренным сообщениям, даже если они в основном пропускаются как уже сохранённые, служебные или bot-сообщения.

### Fixed
- Исправлена остановка `scripts/gk_collector.py` по `Ctrl+C` в backfill-режиме: сигнал теперь прерывает проход по истории и не заставляет ждать завершения полного backfill-цикла или обработки очереди изображений.

## [0.5.39] - 2026-03-07

### Fixed
- Исправлен обычный backfill Group Knowledge: уже сохранённые сообщения теперь пропускаются до этапа question classification, поэтому без `--force` не выполняется лишний LLM-вызов и не запускается повторная обработка тех же сообщений.

## [0.5.38] - 2026-03-07

### Added
- Для Group Knowledge daemon collector добавлен новый режим `--redirect-test-mode`: он продолжает слушать боевые группы, но публикует ответы в отдельную глобическую test group с метаданными источника (группа, отправитель, исходный вопрос, ссылка на сообщение).

### Changed
- Конфигурация `config/gk_groups.json` теперь может хранить глобическую `test_target_group` для перенаправленного тестового режима Group Knowledge.
- `GroupResponder` поддерживает кросс-групповую отправку через Telethon `send_message`, сохраняя логирование и rate limit по реальной боевой группе.

## [0.5.37] - 2026-03-06

### Added
- В `QASearchService` добавлен явный прогрев (`warmup`) для старта daemon-скрипта: предзагрузка BM25-корпуса и прогрев embedding/vector компонентов до запуска listener.
- В `GroupResponder` добавлен метод `preload_search_resources`, а `gk_collector.py` теперь вызывает его при старте и пишет диагностику прогрева.

### Changed
- Для BM25-кэша добавлена версия корпуса (`signature = count, max_id, max_created_at`): кэш теперь перезагружается не только по TTL, но и сразу при изменении Q&A-корпуса в БД.

## [0.5.36] - 2026-03-06

### Changed
- Улучшены диагностические логи BM25 в Group Knowledge: вместо усечённого списка токенов теперь логируются `query_tokens_head`, `query_tokens_tail`, общее число токенов (`query_tokens_total`), исходное число токенов до фильтрации (`raw_tokens_total`), количество удалённых коротких токенов и список/количество удалённых стоп-слов. Это делает прозрачным, почему часть слов запроса не участвует в BM25 ранжировании.

## [0.5.35] - 2026-03-06

### Changed
- В daemon collector режим `--test-mode` теперь всегда запускает встроенный автоответчик в LIVE-режиме (без dry-run), даже если флаг `--live` не передан: ответы отправляются reply-сообщениями в тестовую группу по умолчанию для удобного end-to-end тестирования.
- Лог старта daemon collector теперь показывает фактический режим автоответчика (`LIVE`/`DRY-RUN`) с учётом переопределения test-mode.

### Fixed
- В daemon bridge (`collector_responder`) перед запуском автоответчика добавлено обновление pending-сообщений из БД по `group_id + telegram_message_id`, чтобы в `question_override` попадали актуальные `caption` и `image_description` от очереди обработки изображений.
- Исправлен кейс, когда пользователь отправляет изображение вместе с текстом (до/после или подписью): описание изображения теперь не теряется при склейке и передаётся в RAG-контекст через `full_text`.

## [0.5.33] - 2026-03-06

### Changed
- Поиск по Q&A-парам Group Knowledge переведён на гибридный BM25 + Vector с объединением через Reciprocal Rank Fusion (RRF): лексический BM25-поиск по in-memory корпусу с Russian-нормализацией (лемматизация/стемминг) и векторный поиск через Qdrant объединяются по формуле RRF, обеспечивая более точное ранжирование.
- Добавлены подробные диагностические INFO-логи для каждого шага поиска: BM25 (размер корпуса, top score, query tokens), vector (hits, top score), RRF (позиции и score от каждого метода для каждого кандидата).
- BM25-корпус Q&A-пар кэшируется в памяти с настраиваемым TTL (`GK_BM25_CORPUS_TTL_SECONDS`) для минимизации нагрузки на БД.

### Added
- Настройки гибридного поиска GK: `GK_HYBRID_ENABLED`, `GK_BM25_K1`, `GK_BM25_B`, `GK_RRF_K`, `GK_RU_NORMALIZATION_ENABLED`, `GK_SEARCH_CANDIDATES_PER_METHOD`, `GK_BM25_CORPUS_TTL_SECONDS`.

## [0.5.32] - 2026-03-06

### Changed
- В `test-mode` для сообщений тестовых групп отключено блокирование по `GK_IGNORED_SENDER_IDS`: сообщения из тестовой группы больше не отбрасываются bridge/responder только из-за того, что sender-id находится в ignore-списке для боевого режима.

## [0.5.31] - 2026-03-06

### Changed
- В daemon-пути `gk_collector -> collector_responder -> responder` добавлены дополнительные диагностические `INFO`-логи для определения причины пропуска сообщения (включая случай, когда sender попадает в `GK_IGNORED_SENDER_IDS`) и подтверждения передачи сообщения в bridge/RAG.

## [0.5.30] - 2026-03-06

### Changed
- Для daemon-автоответчика Group Knowledge добавлены дополнительные `INFO`-логи: постановка сообщения в буфер склейки, запуск bridge, старт RAG-поиска и явный лог, если подходящий ответ не найден.
- В документации уточнено, что в `test-mode` запуск RAG происходит после окна склейки `GK_MESSAGE_GROUPING_WINDOW_SECONDS`, поэтому при значении `20` поиск стартует примерно через 20 секунд после получения сообщения.

## [0.5.29] - 2026-03-06

### Changed
- `MessageCollector` теперь пишет в логи timestamp исходного Telegram-сообщения при live-сборе, backfill и автоматическом доборе пропущенных сообщений.

## [0.5.28] - 2026-03-06

### Added
- В `MessageCollector` добавлен автоматический добор пропущенных сообщений при старте daemon collector: перед подпиской на новые события скрипт догружает все сообщения новее последнего сохранённого `telegram_message_id` по каждой группе.

### Changed
- `scripts/gk_collector.py` теперь при обычном запуске сначала выполняет catch-up синхронизацию пропущенных сообщений, а затем поднимает live-listener.

## [0.5.27] - 2026-03-06

### Added
- В `gk_collector.py` добавлен встроенный daemon-автоответчик с флагами `--live`, `--test-mode` и `--collect-only`.
- Добавлен bridge `src/group_knowledge/collector_responder.py`, который склеивает соседние сообщения одного пользователя в один вопрос, включая сценарии `изображение -> текст` и `текст -> изображение`.
- В настройки Group Knowledge добавлены `GK_IGNORED_SENDER_IDS` и `GK_MESSAGE_GROUPING_WINDOW_SECONDS`.

### Changed
- Основной runtime-сценарий автоответчика Group Knowledge перенесён в daemon `scripts/gk_collector.py`.
- `qa_analyzer.py` теперь исключает сообщения sender-id из `GK_IGNORED_SENDER_IDS` до построения цепочек и LLM-inferred анализа.
- `responder.py` теперь игнорирует sender-id из `GK_IGNORED_SENDER_IDS`, чтобы бот не отвечал сам себе.

## [0.5.26] - 2026-03-06

### Added
- Добавлен общий сервис `src/group_knowledge/question_classifier.py` для LLM-классификации сообщений как вопросов.
- В `gk_messages` добавлены поля `is_question`, `question_confidence`, `question_reason`, `question_model_used`, `question_detected_at`, а также SQL-миграция `sql/group_knowledge_question_classification_setup.sql`.

### Changed
- `message_collector.py` теперь классифицирует новые и backfill-сообщения как вопрос/не вопрос при сохранении в БД.
- `qa_analyzer.py` теперь использует сохранённые question-метки как подсказки `QUESTION_HINT` / `NOT_QUESTION_HINT` в LLM-inferred анализе.
- `responder.py` переведён на общий `QuestionClassifierService`, чтобы классификация вопроса была единообразной во всём Group Knowledge.

## [0.5.25] - 2026-03-06

### Added
- В `scripts/gk_responder.py` добавлен `--test-mode`: интерактивный выбор связки `test group -> real group`, после чего автоответчик в daemon-режиме слушает тестовую группу, отвечает в неё же, но логически обрабатывает сообщения как относящиеся к выбранной реальной группе.

### Changed
- В `src/group_knowledge/responder.py` для сообщений без символа `?` добавлена LLM-классификация вопроса: такие сообщения отправляются в модель, чтобы определить, являются ли они техническим вопросом.
- В test mode `GroupResponder` использует маппинг `test_group_id -> real_group_id` для логирования и group-level rate limiting, сохраняя отправку ответа в фактическую тестовую группу.

## [0.5.24] - 2026-03-06

### Added
- В `scripts/gk_delete_group_data.py` добавлен интерактивный режим меню выбора группы (`--interactive`), а также автоматический вход в меню, если `--group-id` не указан.

### Changed
- В интерактивном режиме `gk_delete_group_data.py` теперь запрашивает подтверждение удаления через ввод `yes` после показа dry-run статистики.

## [0.5.23] - 2026-03-06

### Added
- Добавлена утилита `scripts/gk_delete_group_data.py` для безопасной очистки данных Group Knowledge по конкретной группе: dry-run предпросмотр, подтверждение удаления через `--yes`, удаление данных из `gk_messages`, `gk_qa_pairs`, `gk_responder_log`, связанных записей `gk_image_queue` и очистка векторных точек Q&A из `gk_qa_pairs_v1`.

### Changed
- В `src/group_knowledge/database.py` добавлены сервисные функции `get_qa_pair_ids_by_group` и `delete_group_data` для централизованной очистки данных группы и использования в CLI-утилитах.

## [0.5.22] - 2026-03-06

### Added
- В `scripts/gk_analyze.py` добавлен флаг `--all-unprocessed`, который запускает анализ по всем датам, где в `gk_messages` есть сообщения с `processed = 0`, без необходимости вручную указывать `--date` или `--date-range`.

### Changed
- Для `--all-unprocessed` добавлен явный запрет совместного использования с `--force-reanalyze`, чтобы избежать конфликтующей семантики запуска.

## [0.5.21] - 2026-03-06

### Changed
- В thread-based анализе Group Knowledge пересекающиеся nearby-расширенные цепочки теперь не отбрасываются, а объединяются в одну общую цепочку (при overlap не менее 60% относительно меньшей цепочки), чтобы в финальный контекст попадали ответы всех участников обсуждения.

## [0.5.20] - 2026-03-06

### Fixed
- В thread-based анализе Group Knowledge добавлена дедупликация пересекающихся nearby-расширенных цепочек до LLM-валидации: при overlap не менее 60% (относительно меньшей цепочки) теперь обрабатывается только один кандидат — с большим числом сообщений, а при равном размере выбирается более ранний root.

## [0.5.19] - 2026-03-06

### Changed
- В `gk_analyze.py` отключены сокращённые long-аргументы argparse (`allow_abbrev=False`), чтобы частичные ключи вроде `--for` больше не принимались как `--force-reanalyze`.

## [0.5.18] - 2026-03-06

### Changed
- В thread-based анализе Group Knowledge добавлено явное логирование случаев, когда к reply-цепочке подтягиваются дополнительные соседние последовательные сообщения (без прямого reply), с количеством добавленных сообщений.

## [0.5.17] - 2026-03-06

### Changed
- Thread-based анализ Group Knowledge теперь расширяет reply-цепочку соседними последовательными сообщениями от тех же участников беседы, даже если у этих сообщений нет прямой Telegram reply-связи.

## [0.5.16] - 2026-03-06

### Changed
- `gk_responder.py` теперь обрабатывает только сообщения, начинающиеся с команды `/qa` (включая вариант `/qa@BotName ...`). Текст вопроса после команды передаётся в поисковый контур как основной запрос.

## [0.5.15] - 2026-03-06

### Fixed
- В Group Knowledge добавлена защита от дублей Q&A по `question_message_id`: `store_qa_pair` теперь обновляет существующую запись вместо создания новой.

### Added
- Добавлена SQL-миграция `sql/group_knowledge_qa_question_unique.sql` для очистки уже накопленных дублей и установки `UNIQUE`-индекса `uk_question_message_id`.

## [0.5.14] - 2026-03-06

### Added
- В `gk_analyze.py` добавлен флаг `--force-reanalyze`, который принудительно запускает анализ по всем сообщениям за дату (включая уже `processed=1`) для осознанного перерасчёта Q&A-пар.

## [0.5.13] - 2026-03-06

### Changed
- `gk_analyze.py` теперь берёт для анализа только сообщения `processed = 0`, чтобы повторные запуски не пересоздавали уже извлечённые Q&A-пары.

## [0.5.12] - 2026-03-06

### Fixed
- Group Knowledge больше не сохраняет в БД пустые Q&A-пары из некорректных LLM-ответов (в том числе кейсы с `is_valid_qa=false` и/или пустыми `clean_question`/`clean_answer`). Добавлена нормализация и проверка полей перед `store_qa_pair` в thread-based и LLM-inferred потоках.

## [0.5.11] - 2026-03-06

### Changed
- Thread-based валидация Group Knowledge теперь передаёт в LLM-контекст не только текст цепочки, но и timestamp каждого сообщения вместе с именем автора, чтобы модель лучше понимала ход обсуждения.
- При сохранении Q&A-пар Group Knowledge в `question_text` теперь автоматически добавляется gist проблемы с изображения, если исходный вопрос сопровождался описанным скриншотом.

## [0.5.10] - 2026-03-06

### Fixed
- Исправлена автоматическая индексация Group Knowledge в `gk_analyze.py`: Q&A-пары снова корректно сохраняются в отдельную Qdrant-коллекцию через актуальный интерфейс `LocalVectorIndex`, без падения с ошибкой `'<=' not supported between instances of 'str' and 'int'`.
- Исправлен векторный поиск автоответчика Group Knowledge: найденные кандидаты теперь резолвятся обратно в полные Q&A-пары из MySQL по `pair_id`, что совместимо с текущим форматом `vector_search.py`.

## [0.5.9] - 2026-03-06

### Changed
- Для `gk_collector.py` и `gk_responder.py` убраны автоматические retry при `database is locked` у Telethon session-файла. Теперь при lock-конфликте скрипт сразу завершится с понятной ошибкой, а повторный запуск нужно делать вручную.

## [0.5.8] - 2026-03-06

### Fixed
- Запуск `gk_collector.py` и `gk_responder.py` стал устойчивее к временным SQLite-lock конфликтам Telethon session-файла после остановки через `Ctrl+C`: добавлены `busy_timeout`, несколько автоматических retry и более надёжное закрытие клиента при завершении.

## [0.5.7] - 2026-03-06

### Added
- В `gk_collector.py --backfill` добавлен флаг `--force`, который принудительно обновляет уже собранные сообщения, заново скачивает изображения и пересоздаёт их описания через GigaChat.

### Changed
- Обычный backfill Group Knowledge теперь пропускает уже собранные сообщения, чтобы не создавать лишние повторные загрузки изображений и дублирующую очередь на описание.

## [0.5.6] - 2026-03-06

### Fixed
- `gk_collector.py` и `gk_responder.py` больше не переиспользуют `helper_session`; теперь каждый скрипт всегда открывает только свою выделенную Telethon-сессию, что устраняет частые sqlite-ошибки `database is locked` при параллельной работе со скриптами.
- Обновлена документация Group Knowledge: явно указано, что `helper_session` не используется для подсистемы Group Knowledge во избежание lock-конфликтов.

## [0.5.5] - 2026-03-06

### Fixed
- Улучшены сообщения об ошибках авторизации GigaChat: при неверном `GIGACHAT_CREDENTIALS` теперь выводится явная подсказка, что нужен `Authorization key` (Basic key), а не access token или обычный API key.
- Уточнена документация Group Knowledge по настройке `GIGACHAT_CREDENTIALS`.

## [0.5.4] - 2026-03-06

### Changed
- Автоответчик Group Knowledge теперь добавляет в ответ ссылку на сообщение в группе с похожим ранее решённым кейсом.

## [0.5.3] - 2026-03-06

### Changed
- Thread-based анализ Group Knowledge теперь умеет извлекать Q&A не только из одного прямого reply, но и из целых цепочек обсуждения вида «вопрос → уточнение → решение → благодарность», собирая итоговый ответ по всей ветке.

## [0.5.2] - 2026-03-06

### Changed
- В `gk_collector.py --manage-groups` добавлен интерактивный выбор групп из списка доступных Telegram-групп и супергрупп аккаунта, без необходимости вручную вводить ID.
