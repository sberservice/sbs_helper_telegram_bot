# Group Knowledge — Подсистема добычи знаний из групп

## Обзор

**Group Knowledge** — подсистема для автоматического извлечения технических знаний из переписок в Telegram-группах техподдержки. Работает как набор standalone-скриптов, не зависящих от основного бота.

### Что делает

1. **Коллектор** — слушает сообщения в настроенных Telegram-группах через Telethon, сохраняет текст/подписи/изображения в БД.
2. **Обработчик изображений** — описывает скриншоты через GigaChat Vision API, привязывая текстовое описание к сообщениям.
3. **Анализатор** — извлекает пары вопрос-ответ из собранных сообщений: по reply-to связям и полным цепочкам обсуждения (thread-based), а также через LLM-анализ контекста (LLM-inferred).
4. **Автоответчик** — в daemon-режиме встроен в коллектор: обнаруживает вопросы в группах и отвечает из накопленной базы Q&A-пар через RAG (с dry-run режимом по умолчанию).
	При ответе также прикладывает ссылку на похожее обсуждение в группе, из которого была найдена релевантная Q&A-пара.

### Архитектура

```
scripts/
├── gk_collector.py        ← Standalone: сбор сообщений + обработка изображений + встроенный автоответчик daemon
├── gk_analyze.py          ← CLI: извлечение Q&A-пар + индексация в Qdrant
└── gk_responder.py        ← Standalone: legacy-автоответчик для отдельного запуска

src/group_knowledge/
├── __init__.py
├── settings.py            ← Константы и метаданные модуля
├── models.py              ← Модели данных (GroupMessage, QAPair, ...)
├── database.py            ← Слой работы с MySQL
├── collector_responder.py ← Склейка соседних сообщений и мост к автоответчику
├── image_processor.py     ← Скачивание + описание изображений (GigaChat)
├── message_collector.py   ← Telethon listener + управление группами
├── qa_analyzer.py         ← Извлечение Q&A пар (thread + LLM)
├── qa_search.py           ← Гибридный поиск по Q&A базе
├── question_classifier.py ← Общая LLM-классификация сообщений как вопрос/не вопрос
└── responder.py           ← Автоответчик с rate limiting
```

## Быстрый старт

### 1. Настройка окружения

Добавьте в `.env`:

```env
# GigaChat для описания изображений
# Здесь нужен Authorization key / Basic key из кабинета Sber Developers,
# а не access token и не обычный API key.
GIGACHAT_CREDENTIALS=your_gigachat_authorization_key
GIGACHAT_SCOPE=GIGACHAT_API_PERS  # или GIGACHAT_API_CORP

# Telethon (если ещё не настроен)
TELETHON_API_ID=12345
TELETHON_API_HASH=abc123def456
```

### 2. Создание таблиц в БД

```bash
mysql -u root -p archie_db < sql/group_knowledge_setup.sql

# (для существующих установок) добавить поля классификации вопроса в gk_messages
mysql -u root -p archie_db < sql/group_knowledge_question_classification_setup.sql

# (для существующих установок) добавить поля качества Q&A-пары в gk_qa_pairs
mysql -u root -p archie_db < sql/gk_qa_pairs_quality_fields_setup.sql

# (опционально) очистка старых дублей и защита от новых
mysql -u root -p archie_db < sql/group_knowledge_qa_question_unique.sql
```

### 3. Настройка групп

```bash
python scripts/gk_collector.py --manage-groups
```

Интерактивное CLI для добавления/удаления групп. Группы можно выбрать прямо из списка
доступных групп и супергрупп аккаунта Telegram. Конфигурация сохраняется в `config/gk_groups.json`.

Группы можно временно отключить через админ-панель (Process Manager → Группы) без удаления
из конфига. Отключённые группы имеют `"disabled": true` в JSON и исключаются из сбора сообщений
и автоответов. Для применения изменений требуется перезапуск коллектора.

> Важно: при первой авторизации Telethon код подтверждения обычно приходит в приложение Telegram
> (служебный чат `Telegram`), а не по SMS. Номер телефона вводите в международном формате,
> например `+79991234567`.

> Важно для GigaChat: в `GIGACHAT_CREDENTIALS` нужно указывать именно `Authorization key`
> (Basic key) из кабинета Sber Developers. Если указать access token или неподходящий ключ,
> GigaChat вернёт ошибку `Authorization error: header is incorrect`.

### 4. Запуск коллектора

```bash
# Слушатель новых сообщений + встроенный автоответчик в dry-run режиме
python scripts/gk_collector.py

# Реально отправлять ответы из встроенного автоответчика
python scripts/gk_collector.py --live

# Только сборщик без встроенного автоответчика
python scripts/gk_collector.py --collect-only

# Test mode: слушать test group, но использовать базу знаний real group
# (в test-mode ответы отправляются реально, даже без --live)
python scripts/gk_collector.py --test-mode

# Redirect test mode: слушать боевые группы, но отправлять ответы в отдельную test group
# (в redirect-test-mode ответы тоже отправляются реально, даже без --live)
python scripts/gk_collector.py --redirect-test-mode

# Или загрузка истории за 7 дней
python scripts/gk_collector.py --backfill --days 7

# Принудительно пересобрать уже загруженные сообщения,
# заново скачать изображения и пересоздать их описания
python scripts/gk_collector.py --backfill --days 7 --force
```

Флаг `--force` полезен, если нужно повторно прогнать GigaChat по изображениям,
перекачать медиафайлы или обновить уже собранные сообщения без ручной очистки БД.
Без `--force` обычный backfill теперь полностью пропускает уже сохранённые сообщения:
для них повторно не запускается LLM-классификация вопроса и не ставится в очередь
повторная обработка изображений.
Во время backfill в логах теперь показывается прогресс не только по сохранённым,
но и по уже просмотренным сообщениям: видно, сколько сообщений просмотрено,
сохранено и пропущено по причинам `existing/service/bots`. Это особенно полезно,
когда большая часть истории уже есть в БД и backfill в основном делает skip.
`Ctrl+C` в backfill-режиме теперь останавливает проход по истории и последующую
обработку очереди изображений без долгого ожидания завершения полного цикла.

В daemon-режиме `gk_collector.py` теперь выполняет две задачи одновременно:
- сохраняет новые сообщения и медиа в БД;
- запускает встроенный автоответчик, если не указан `--collect-only`.

Перед стартом live-listener коллектор также автоматически добирает сообщения,
которые успели появиться в группе с прошлого запуска. Для этого он берёт из БД
последний сохранённый `telegram_message_id` по каждой группе и догружает все более
новые сообщения. Если группа ещё ни разу не собиралась, автоматический добор не
выполняется — для первичного наполнения используйте `--backfill`.

В логах collector для каждого сохранённого сообщения теперь дополнительно показывается
исходный timestamp самого сообщения, чтобы было видно фактическое время его отправки,
а не только время записи в БД.

Для обычных сообщений daemon ждёт короткое окно склейки и объединяет соседние сообщения
одного и того же пользователя в один запрос. Это позволяет корректно обработать сценарии,
когда пользователь сначала отправляет изображение, а через несколько секунд — текст,
или наоборот. Команда `/qa ...` обрабатывается сразу, без ожидания окна склейки.

### 5. Анализ и извлечение Q&A

```bash
# Анализ за вчера (по умолчанию)
python scripts/gk_analyze.py

# Конкретная дата
python scripts/gk_analyze.py --date 2024-01-15

# Диапазон дат
python scripts/gk_analyze.py --date-range 2024-01-10 2024-01-15

# Все даты с необработанными сообщениями (processed = 0)
python scripts/gk_analyze.py --all-unprocessed

# Только индексация в Qdrant
python scripts/gk_analyze.py --index

# Полностью пересобрать QA-векторный индекс (удалить старые QA-векторы и переиндексировать approved-пары)
python scripts/gk_analyze.py --rebuild-vector-index

# Принудительный переанализ (включая уже processed сообщения)
python scripts/gk_analyze.py --date 2024-01-15 --force-reanalyze

# Полный rebuild всех Q&A-пар по всем датам выбранных групп
python scripts/gk_analyze.py --all-dates --rebuild-pairs
```

`gk_analyze.py` по умолчанию анализирует только сообщения с `processed = 0`.
Это делает повторные запуски идемпотентными: уже обработанные сообщения не будут
повторно извлекаться в новые Q&A-пары.
Если нужен повторный разбор исторических данных, используйте флаг `--force-reanalyze`.
Для прохода по всем накопленным необработанным сообщениям без ручного указания даты используйте `--all-unprocessed`.
Если нужно пересобрать только векторный слой без повторного анализа сообщений,
используйте `--rebuild-vector-index`: режим удаляет QA-векторы для текущих approved-пар,
сбрасывает `vector_indexed` и запускает повторную индексацию из `gk_qa_pairs`.
Если после изменения логики извлечения нужно полностью пересобрать Q&A-пары, используйте
`--all-dates --rebuild-pairs`: режим удалит старые `gk_qa_pairs`, очистит их vector-точки
и заново прогонит анализ по всем датам, где у выбранных групп есть сообщения.

Новые сообщения при сборе теперь дополнительно классифицируются как вопрос/не вопрос.
Для сообщений без символа `?` коллектор отправляет текст в LLM, сохраняет результат в `gk_messages`
(`is_question`, `question_confidence`, `question_reason`, `question_model_used`, `question_detected_at`),
а анализатор позже использует эти метки как подсказки `QUESTION_HINT` / `NOT_QUESTION_HINT` во время LLM-inferred анализа
и thread-chain валидации. Если `is_question = 1` и `question_confidence >= GK_ANALYSIS_QUESTION_CONFIDENCE_THRESHOLD`
(по умолчанию `0.90`), такое сообщение считается явным вопросом цепочки даже если reply-root был служебным или вводным сообщением.
Для LLM-классификации вопроса также учитывается контекст медиа: если текст пришёл как `caption` к изображению,
это передаётся в prompt как сильный сигнал возможного вопроса. При наличии `image_description` описание изображения
тоже добавляется в метаданные классификатора, чтобы короткие подписи вроде `что с этим` точнее распознавались как запрос помощи.
Сообщения от sender-id из `GK_IGNORED_SENDER_IDS` анализатор исключает до построения цепочек,
чтобы служебные сообщения бота не попадали в Q&A-пары.

Для `llm_inferred` есть два независимых флага:
- `GK_GENERATE_LLM_INFERRED_QA_PAIRS` — управляет самой генерацией таких пар в `gk_analyze`.
  При значении `0` фаза LLM-inferred extraction полностью пропускается, и новые `llm_inferred`
  пары не создаются в БД.
- `GK_INCLUDE_LLM_INFERRED_ANSWERS` — управляет использованием уже существующих `llm_inferred`
  пар в поисковом автоответчике.

Если `GK_INCLUDE_LLM_INFERRED_ANSWERS=0`, RAG-ответы строятся только по `thread_reply` парам,
а `llm_inferred` пары (если они уже были созданы ранее) не используются в BM25/vector поиске
и в финальном LLM-контексте ответа.

В поиске GK защищённые термины для токенизации формируются как объединение:
- `fixed_term` со статусом `approved`;
- `acronym` со статусом `approved` и `confidence >= GK_ACRONYMS_MIN_CONFIDENCE`.

Это позволяет сохранять короткие аббревиатуры (например, `фн`, `чз`) в BM25-пайплайне,
где обычные короткие токены обычно отфильтровываются. Поддерживаются как однословные,
так и составные варианты (например, `эвотор 6`). Составные термины перед токенизацией
канонизируются во внутренний вид с `_` (например, `эвотор_6`), чтобы не теряться
в BM25/spellcheck пайплайне.

Флаг `GK_RAG_IMAGE_GIST_ENABLED` управляет добавлением `image_description` в текст вопроса **только для RAG-слоя**
(BM25-корпус и векторная индексация). При этом в `gk_qa_pairs.question_text` сохраняется чистый вопрос без добавленного gist,
что упрощает хранение и ручную валидацию пар в БД.

### 6. Запуск автоответчика

```bash
# Основной рекомендованный режим: встроенный автоответчик внутри коллектора
python scripts/gk_collector.py

# Live режим — реально отправляет ответы
python scripts/gk_collector.py --live

# Test mode — выбрать test group и real group через интерактивное меню
python scripts/gk_collector.py --test-mode

# Redirect test mode — слушать боевые группы и пересылать ответы в одну test group
python scripts/gk_collector.py --redirect-test-mode
```

Встроенный автоответчик в `gk_collector.py` работает так:
- команда `/qa ...` всегда обрабатывается сразу и считается вопросом принудительно;
- обычные сообщения сначала буферизуются по паре `(group_id, sender_id)` на короткое окно склейки;
- текст и изображения одного пользователя объединяются в один запрос;
- если явного `?` нет, используется сохранённая или онлайн LLM-классификация вопроса.

Пример запроса через команду:

```text
/qa как исправить ошибку 1001 на терминале?
```

Сообщения без префикса `/qa` не игнорируются полностью: daemon collector может ответить и на них,
если после склейки и классификации сообщение распознано как технический вопрос.

В `--test-mode` встроенный автоответчик работает как daemon: после интерактивного выбора связки
`test group -> real group` он постоянно слушает новые сообщения в тестовой группе,
но логически считает их сообщениями реальной группы.

`--test-mode` принудительно отключает dry-run для встроенного автоответчика,
поэтому ответы отправляются reply-сообщениями в тестовую группу даже без флага `--live`.

В `--redirect-test-mode` daemon продолжает слушать обычные боевые группы из `config/gk_groups.json`,
сохраняет и классифицирует их сообщения как обычно, но вместо reply в исходный чат отправляет
сгенерированный ответ в одну отдельную тестовую группу. При первом запуске режим попросит выбрать
эту группу интерактивно и сохранит её в `config/gk_groups.json` в поле `test_target_group`.
Дальше будет использоваться сохранённая test group, пока вы не измените JSON вручную.

Сообщение в test group содержит:
- источник (название и id боевой группы);
- отправителя исходного вопроса;
- ссылку на исходное сообщение, если её можно построить;
- оригинальный вопрос;
- сгенерированный ответ.

`--redirect-test-mode` тоже принудительно отключает dry-run, поэтому ответы реально публикуются
в тестовую группу даже без флага `--live`.

Для `test-mode` фильтр `GK_IGNORED_SENDER_IDS` на сообщения тестовой группы не применяется,
чтобы можно было полноценно проверять RAG на своих сообщениях даже если ваш sender-id
входит в список игнорируемых для боевого режима.

Важно: в `test-mode` и обычном daemon-режиме RAG-поиск стартует не мгновенно, а после окна
склейки `GK_MESSAGE_GROUPING_WINDOW_SECONDS`. Если сейчас оно равно `20`, то для одиночного
сообщения запуск поиска будет виден примерно через 20 секунд после лога о сохранении сообщения.
В логах также появятся отдельные записи о постановке сообщения в буфер, запуске bridge и старте RAG.
Перед запуском bridge дополнительно перечитывает сообщения из БД по `group_id + telegram_message_id`,
чтобы при склейке использовать актуальные `caption` и `image_description`,
которые могли появиться асинхронно после обработки изображения в очереди.
При старте daemon также выполняется прогрев поиска: предзагрузка BM25-корпуса
и прогрев vector/embedding компонентов, чтобы первый реальный запрос не тратил время
на холодную инициализацию моделей.
Для офлайн-эксплуатации рекомендуется заранее прогреть embedding-модель командой
`python scripts/rag_ops.py preload-embeddings` и использовать постоянный кэш
`AI_RAG_VECTOR_EMBEDDING_CACHE_DIR`.
Если включены `AI_RAG_VECTOR_EMBEDDING_OFFLINE=1` и
`AI_RAG_VECTOR_EMBEDDING_FAIL_FAST=1`, daemon завершится на старте с ошибкой,
если модель не найдена в локальном кэше.
Если после сохранения сообщения виден лог `пропуск — sender в GK_IGNORED_SENDER_IDS`,
значит sender-id пользователя попал в список игнорируемых и автоответчик сознательно
не запускается для таких сообщений.

Скрипт `gk_responder.py` сохранён для отдельного legacy-запуска, но основной эксплуатационный
сценарий теперь проходит через daemon `gk_collector.py`.

Дополнительно у `gk_collector.py` есть режим заполнения question-классификации для уже собранных сообщений:

```bash
# Заполнить is_question только для сообщений, где поле ещё NULL
python scripts/gk_collector.py --fill-missing-is-question

# Только за последние 30 дней
python scripts/gk_collector.py --fill-missing-is-question --fill-days 30

# Только для одной группы и ограниченным батчем
python scripts/gk_collector.py --fill-missing-is-question --group-id -1001234567890 --fill-limit 500
```

Этот режим не требует Telethon-подключения: он читает сообщения из БД, находит записи с `is_question IS NULL`,
прогоняет их через ту же логику классификации, что и live/backfill-сбор, и сохраняет результат обратно в `gk_messages`.

### 7. Очистка данных по группе

```bash
# Предпросмотр (dry-run): показать, сколько записей будет удалено
python scripts/gk_delete_group_data.py --group-id -1001234567890

# Интерактивное меню выбора группы (если --group-id не передан, режим включается автоматически)
python scripts/gk_delete_group_data.py --interactive

# Реальное удаление данных группы
python scripts/gk_delete_group_data.py --group-id -1001234567890 --yes

# Реальное удаление без очистки векторных точек
python scripts/gk_delete_group_data.py --group-id -1001234567890 --yes --no-vector-cleanup
```

Утилита удаляет данные группы из `gk_messages`, `gk_qa_pairs`, `gk_responder_log`,
связанные записи `gk_image_queue`, а также по умолчанию очищает точки Q&A в коллекции
`gk_qa_pairs_v1` в Qdrant.
В интерактивном режиме дополнительно запрашивается подтверждение удаления через ввод `yes`.

### Основные настройки (`config/ai_settings.py`)

### Telethon-сессии

Коллектор и standalone-автоответчик используют отдельные Telethon-сессии:
- `gk_collector_session` — для коллектора
- `gk_responder_session` — для legacy-скрипта автоответчика

Сессия `helper_session` намеренно не переиспользуется: это снижает риск sqlite-конфликтов Telethon вида `database is locked`, если параллельно работает другой скрипт.

Если предыдущий запуск был остановлен через `Ctrl+C`, следующий старт всё ещё может упереться в временный lock session-файла. В этом случае Group Knowledge завершится с понятной ошибкой, и запуск нужно повторить вручную после освобождения файла.

При первом запуске потребуется авторизация по номеру телефона.
Код подтверждения обычно приходит в приложение Telegram, а если на аккаунте включена
двухэтапная аутентификация, после кода потребуется ввести облачный пароль.

## Алгоритм извлечения Q&A

### Thread-based (Phase 1)

1. Найти все reply-цепочки за день и собрать полные обсуждения от корневого вопроса до финального решения.
2. Включить в цепочку уточняющие вопросы, промежуточные ответы и служебные reply-сообщения.
3. Дополнительно подтянуть соседние (nearby) последовательные сообщения от тех же участников цепочки, даже если между ними нет Telegram reply-связи.
  Для составных вопросов от одного автора допускается присоединение нескольких подряд идущих question-сообщений (в коротком временном окне), чтобы не терялась первая часть вопроса.
4. После nearby-расширения добрать reply-потомков для сообщений, которые были добавлены без явной reply-связи на первом шаге, чтобы восстановить полную ветку обсуждения.
5. Если после nearby-расширения несколько кандидатов сильно пересекаются (overlap не менее 60% относительно меньшей цепочки), объединить их в одну цепочку, чтобы собрать ответы всех участников в общий контекст; root у объединённой цепочки выбирается более ранний.
6. Передать в LLM не только текст сообщений, но и время каждого сообщения, а также имя автора, чтобы модели было проще понять ход обсуждения.
7. Отделить итоговое решение от благодарностей и прочего шума.
8. При сохранении вопроса в БД дополнительно включить gist проблемы с изображения, если исходный вопрос сопровождался скриншотом или фото с описанием от GigaChat.
9. Валидировать всю цепочку через LLM: является ли она полезным техническим Q&A?
10. Если валидна — сохранить одну итоговую очищенную пару вопрос-ответ по всей цепочке.

### LLM-inferred (Phase 2)

1. Собрать сообщения, не вошедшие в thread-пары.
2. Разбить на батчи по `GK_ANALYSIS_BATCH_SIZE`.
3. Отправить батч в LLM с промптом поиска семантических Q&A пар.
4. Сохранить найденные пары.

### Индексация (Phase 3)

1. Найти новые неиндексированные пары.
2. Создать эмбеддинг (вопрос + ответ) через локальный embedding provider.
3. Сохранить каждую пару как отдельный векторный документ в Qdrant коллекцию `gk_qa_pairs_v1`.
4. При поиске по этой коллекции затем подтянуть полную Q&A-пару из MySQL по `pair_id`, чтобы автоответчик работал на актуальных данных из БД.

### Гибридный поиск (BM25 + Vector + RRF)

При поступлении вопроса от пользователя `QASearchService` выполняет два параллельных поиска
и объединяет результаты через **Reciprocal Rank Fusion (RRF)**:

#### Шаг 1 — BM25 (лексический)

1. Загрузить все одобренные Q&A-пары из MySQL (`gk_qa_pairs WHERE approved = 1 AND expert_status != 'rejected'`).
2. Токенизировать каждую пару (`question_text + answer_text`) с Russian-нормализацией:
  - Regex-извлечение слов, фильтрация коротких (< 3 символов, кроме защищённых терминов и approved-аббревиатур с confidence-порогом, например: `nfc`, `pos`, `ккт`, `фн`, `чз`)
   - Удаление стоп-слов (`что`, `как`, `это`, `для` и т.п.)
   - Лемматизация через pymorphy3 + стемминг через snowballstemmer.
3. Вычислить BM25-score запроса относительно каждой пары (`rank_bm25.BM25Okapi` с параметрами `k1=1.5`, `b=0.75`).
4. Вернуть top-K кандидатов, отсортированных по BM25-score.

Корпус кэшируется в памяти с настраиваемым TTL (`GK_BM25_CORPUS_TTL_SECONDS`, по умолчанию 300с).
Дополнительно используется сигнатура версии корпуса (`count`, `max_id`, `max_created_at`):
если в БД появляются новые approved Q&A-пары, BM25-кэш перезагружается сразу,
даже если TTL ещё не истёк.

#### Шаг 2 — Vector (семантический)

1. Вычислить embedding запроса через `LocalEmbeddingProvider` (модель `BAAI/bge-m3`).
2. Найти ближайшие вектора в Qdrant-коллекции `gk_qa_pairs_v1` по cosine similarity.
3. Поднять полные Q&A-пары из MySQL по `pair_id` из метаданных вектора.

#### Шаг 3 — RRF-слияние

Результаты двух методов объединяются через **Reciprocal Rank Fusion**:

```
RRF_score(d) = Σ 1 / (k + rank_i(d))
```

где `k = 60` (константа по Cormack et al.), `rank_i(d)` — позиция пары `d` в ранжированном
списке метода `i` (1-based).

Пары, присутствующие в обоих списках, получают более высокий RRF-score, что обеспечивает
робастное ранжирование даже при различных шкалах score (BM25 raw vs cosine 0-1).

#### Диагностические логи

На уровне INFO выводится полная таблица RRF-результатов:

```
GK BM25: query_tokens_head=['ошибк', 'продаж', 'изображен', ...]
         query_tokens_tail=['печат', 'кассов', 'чек']
         query_tokens_total=24 raw_tokens_total=31
         removed_short_tokens=2 removed_stopwords=['при', 'почему', ...]
         removed_stopwords_count=5 corpus_size=662 scored_count=521 top_score=67.6739
```

Лог BM25 специально показывает не только «голову» токенов, но и «хвост», а также
удалённые слова, чтобы было видно, почему часть исходной фразы не попала в итоговые `query_tokens`.

```
GK RRF-поиск: query="Как включить NFC?" top_results=5
  #1 pair=17 rrf=0.032787 bm25_rank=1 bm25_score=4.2300 vec_rank=2 vec_score=0.8700 q="Как включить NFC на терминале?"
  #2 pair=42 rrf=0.031250 bm25_rank=3 bm25_score=3.1000 vec_rank=1 vec_score=0.9100 q="Проблема с NFC модулем..."
  #3 pair=8  rrf=0.015873 bm25_rank=— bm25_score=0.0000 vec_rank=3 vec_score=0.8200 q="NFC антенна не работает..."
```

## Автоответчик

### Эвристика определения вопросов

Сообщение считается вопросом, если:
- Содержит `?`
- Начинается с вопросительного слова (как, почему, где, ...)
- Содержит паттерн вроде «не работает», «что делать», «подскажите»
- Или было определено общей LLM-классификацией как технический вопрос без явного `?`

### Склейка соседних сообщений

В daemon collector соседние сообщения одного пользователя в одной группе буферизуются на
`GK_MESSAGE_GROUPING_WINDOW_SECONDS` секунд и затем объединяются в один запрос. Если в окне
есть сообщение только с изображением без текста, в объединённый вопрос добавляется маркер
`[Пользователь приложил изображение без подписи]`.

### Порог уверенности

Ответ отправляется только при `confidence >= GK_RESPONDER_CONFIDENCE_THRESHOLD` (0.7).

### Rate limiting

- Per-user: до 5 ответов за 120 секунд
- Per-group: до 30 ответов за 300 секунд

### Dry-run по умолчанию

По умолчанию встроенный автоответчик работает в режиме dry-run — формирует ответы и логирует их, но не отправляет в группу. Для реального режима используйте флаг `--live`.

Исключение: в `--test-mode` dry-run автоматически отключается для удобной проверки полного цикла ответов в тестовой группе.

### Ссылка на похожий кейс

При успешном ответе автоответчик добавляет ссылку на исходное сообщение в группе, где уже обсуждалась похожая проблема. Обычно это ссылка на сообщение с решением или на наиболее релевантный ответ в найденной цепочке.

## Устранение неполадок

### Миграция группы (изменение ID)

Telegram может автоматически мигрировать обычную группу (Chat) в супергруппу (Channel/Supergroup) при изменении определённых настроек группы (права администраторов, история чата, и т.п.). При миграции **ID группы необратимо меняется**.

**Признаки проблемы:**
- Коллектор перестал собирать сообщения из конкретной группы
- В логах появляются сообщения `Сообщение из неотслеживаемой группы`
- При `resolve_group_ids` выводится предупреждение о несоответствии ID

**Что делает система автоматически:**
- При старте коллектор проверяет каждую группу через Telegram API (`resolve_group_ids`) и обновляет ID в памяти, если обнаружена миграция
- Во время работы перехватывается событие `MessageActionChatMigrateTo` и ID обновляется на лету

**Что нужно сделать вручную:**
1. Обновить ID в `config/gk_groups.json` на новый (будет показан в логах при старте)
2. При необходимости обновить данные в БД

**Формат ID:**
- Обычная группа (Chat): `-XXXXXXXXXX` (например, `-5197204255`)
- Супергруппа/канал (Channel): `-100XXXXXXXXXX` (например, `-1003849210429`)

### Сброс сессии Telethon

Если сессия Telethon повреждена или содержит устаревший кэш сущностей, может потребоваться её полный сброс.

**Когда требуется сброс:**
- Telethon выдаёт ошибки авторизации или кэша
- Группы не распознаются даже после обновления конфига
- Ошибки вида `PeerIdInvalidError` или `ChannelPrivateError` при доступе к группам

**Порядок действий:**

```bash
# 1. Остановить все процессы, использующие сессию
# (коллектор, автоответчик, the_helper)

# 2. Удалить файлы сессии (для коллектора)
rm gk_collector_session.session gk_collector_session.session-shm gk_collector_session.session-wal

# Для автоответчика (если используется отдельная сессия)
rm gk_responder_session.session

# Для The Helper
rm helper_session.session helper_session.session-journal

# Для синхронизации участников
rm chat_sync_session.session

# 3. Запустить процесс заново — Telethon попросит авторизацию
python scripts/gk_collector.py --live
# Ввести номер телефона и код из Telegram
```

> **Внимание:** после сброса сессии потребуется повторная авторизация через код из Telegram. Убедитесь, что у вас есть доступ к телефону, привязанному к аккаунту.
