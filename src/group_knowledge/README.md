# Group Knowledge — Подсистема добычи знаний из групп

## Обзор

**Group Knowledge** — подсистема для автоматического извлечения технических знаний из переписок в Telegram-группах техподдержки. Работает как набор standalone-скриптов, не зависящих от основного бота.

### Что делает

1. **Коллектор** — слушает сообщения в настроенных Telegram-группах через Telethon, сохраняет текст/подписи/изображения в БД.
2. **Обработчик изображений** — описывает скриншоты через GigaChat Vision API, привязывая текстовое описание к сообщениям.
3. **Анализатор** — извлекает пары вопрос-ответ из собранных сообщений: по reply-to связям и полным цепочкам обсуждения (thread-based), а также через LLM-анализ контекста (LLM-inferred).
4. **Автоответчик** — обнаруживает вопросы в группах и отвечает из накопленной базы Q&A-пар через RAG (с dry-run режимом по умолчанию).
	При ответе также прикладывает ссылку на похожее обсуждение в группе, из которого была найдена релевантная Q&A-пара.

### Архитектура

```
scripts/
├── gk_collector.py        ← Standalone: сбор сообщений + обработка изображений
├── gk_analyze.py          ← CLI: извлечение Q&A-пар + индексация в Qdrant
└── gk_responder.py        ← Standalone: автоответчик (dry-run / live)

src/group_knowledge/
├── __init__.py
├── settings.py            ← Константы и метаданные модуля
├── models.py              ← Модели данных (GroupMessage, QAPair, ...)
├── database.py            ← Слой работы с MySQL
├── image_processor.py     ← Скачивание + описание изображений (GigaChat)
├── message_collector.py   ← Telethon listener + управление группами
├── qa_analyzer.py         ← Извлечение Q&A пар (thread + LLM)
├── qa_search.py           ← Гибридный поиск по Q&A базе
└── responder.py           ← Автоответчик с rate limiting
```

## Быстрый старт

### 1. Настройка окружения

Добавьте в `.env`:

```env
# GigaChat для описания изображений
# Здесь нужен Authorization key / Basic key из кабинета Sber Developers,
# а не access token и не обычный API key.
GIGACHAT_CREDENTIALS=your_gigachat_authorization_key
GIGACHAT_SCOPE=GIGACHAT_API_PERS  # или GIGACHAT_API_CORP

# Telethon (если ещё не настроен)
TELETHON_API_ID=12345
TELETHON_API_HASH=abc123def456
```

### 2. Создание таблиц в БД

```bash
mysql -u root -p archie_db < sql/group_knowledge_setup.sql
```

### 3. Настройка групп

```bash
python scripts/gk_collector.py --manage-groups
```

Интерактивное CLI для добавления/удаления групп. Группы можно выбрать прямо из списка
доступных групп и супергрупп аккаунта Telegram. Конфигурация сохраняется в `config/gk_groups.json`.

> Важно: при первой авторизации Telethon код подтверждения обычно приходит в приложение Telegram
> (служебный чат `Telegram`), а не по SMS. Номер телефона вводите в международном формате,
> например `+79991234567`.

> Важно для GigaChat: в `GIGACHAT_CREDENTIALS` нужно указывать именно `Authorization key`
> (Basic key) из кабинета Sber Developers. Если указать access token или неподходящий ключ,
> GigaChat вернёт ошибку `Authorization error: header is incorrect`.

### 4. Запуск коллектора

```bash
# Слушатель новых сообщений
python scripts/gk_collector.py

# Или загрузка истории за 7 дней
python scripts/gk_collector.py --backfill --days 7

# Принудительно пересобрать уже загруженные сообщения,
# заново скачать изображения и пересоздать их описания
python scripts/gk_collector.py --backfill --days 7 --force
```

Флаг `--force` полезен, если нужно повторно прогнать GigaChat по изображениям,
перекачать медиафайлы или обновить уже собранные сообщения без ручной очистки БД.

### 5. Анализ и извлечение Q&A

```bash
# Анализ за вчера (по умолчанию)
python scripts/gk_analyze.py

# Конкретная дата
python scripts/gk_analyze.py --date 2024-01-15

# Диапазон дат
python scripts/gk_analyze.py --date-range 2024-01-10 2024-01-15

# Только индексация в Qdrant
python scripts/gk_analyze.py --index
```

### 6. Запуск автоответчика

```bash
# Dry-run (по умолчанию) — только логирует ответы
python scripts/gk_responder.py

# Live режим — реально отправляет ответы
python scripts/gk_responder.py --live
```

## Конфигурация

### Основные настройки (`config/ai_settings.py`)

| Параметр | Умолчание | Описание |
|----------|-----------|----------|
| `GIGACHAT_CREDENTIALS` | `.env` | API-ключ GigaChat |
| `GIGACHAT_MODEL` | `GigaChat-Pro` | Модель GigaChat для описания изображений |
| `GK_IMAGE_STORAGE_PATH` | `./data/group_knowledge/images` | Путь хранения скачанных изображений |
| `GK_DRY_RUN` | `True` | Глобальный dry-run для автоответчика |
| `GK_RESPONDER_CONFIDENCE_THRESHOLD` | `0.7` | Минимальная уверенность для ответа |
| `GK_QA_VECTOR_COLLECTION` | `gk_qa_pairs_v1` | Коллекция Qdrant для Q&A пар |
| `GK_RESPONDER_TOP_K` | `5` | Число Q&A пар для генерации ответа |
| `GK_ANALYSIS_BATCH_SIZE` | `50` | Размер батча для LLM-анализа |

### Rate limiting (`src/common/constants/sync.py`)

| Параметр | Умолчание | Описание |
|----------|-----------|----------|
| `GK_RATE_LIMIT_USER_MAX` | `5` | Макс. ответов на пользователя в окне |
| `GK_RATE_LIMIT_USER_WINDOW` | `120` с | Окно rate limit для пользователя |
| `GK_RATE_LIMIT_GROUP_MAX` | `30` | Макс. ответов на группу в окне |
| `GK_RATE_LIMIT_GROUP_WINDOW` | `300` с | Окно rate limit для группы |

### Telethon-сессии

Коллектор и автоответчик используют отдельные Telethon-сессии:
- `gk_collector_session` — для коллектора
- `gk_responder_session` — для автоответчика

Сессия `helper_session` намеренно не переиспользуется: это снижает риск sqlite-конфликтов Telethon вида `database is locked`, если параллельно работает другой скрипт.

Если предыдущий запуск был остановлен через `Ctrl+C`, следующий старт всё ещё может упереться в временный lock session-файла. В этом случае Group Knowledge завершится с понятной ошибкой, и запуск нужно повторить вручную после освобождения файла.

При первом запуске потребуется авторизация по номеру телефона.
Код подтверждения обычно приходит в приложение Telegram, а если на аккаунте включена
двухэтапная аутентификация, после кода потребуется ввести облачный пароль.

## Структура БД

### gk\_messages

Собранные сообщения из групп.

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | BIGINT | Внутренний ID |
| `telegram_message_id` | BIGINT | ID сообщения в Telegram |
| `group_id` | BIGINT | ID группы |
| `sender_id` | BIGINT | ID отправителя |
| `message_text` | TEXT | Текст сообщения |
| `caption` | TEXT | Подпись к медиа |
| `has_image` | TINYINT | Есть ли изображение |
| `image_description` | TEXT | Описание изображения от GigaChat |
| `reply_to_message_id` | BIGINT | ID сообщения-ответа |
| `processed` | TINYINT | Обработано анализатором |

### gk\_qa\_pairs

Извлечённые пары вопрос-ответ.

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | BIGINT | Внутренний ID |
| `question_text` | TEXT | Текст вопроса |
| `answer_text` | TEXT | Текст ответа |
| `extraction_type` | ENUM | `thread_reply` / `llm_inferred` |
| `confidence` | FLOAT | Уверенность (0.0–1.0) |
| `approved` | TINYINT | Одобрена (по умолчанию 1) |
| `indexed` | TINYINT | Проиндексирована в Qdrant |

### gk\_image\_queue

Очередь обработки изображений.

### gk\_responder\_log

Лог ответов автоответчика (включая dry-run).

## Алгоритм извлечения Q&A

### Thread-based (Phase 1)

1. Найти все reply-цепочки за день и собрать полные обсуждения от корневого вопроса до финального решения.
2. Включить в цепочку уточняющие вопросы, промежуточные ответы и служебные reply-сообщения.
3. Отделить итоговое решение от благодарностей и прочего шума.
4. Валидировать всю цепочку через LLM: является ли она полезным техническим Q&A?
5. Если валидна — сохранить одну итоговую очищенную пару вопрос-ответ по всей цепочке.

### LLM-inferred (Phase 2)

1. Собрать сообщения, не вошедшие в thread-пары.
2. Разбить на батчи по `GK_ANALYSIS_BATCH_SIZE`.
3. Отправить батч в LLM с промптом поиска семантических Q&A пар.
4. Сохранить найденные пары.

### Индексация (Phase 3)

1. Найти новые неиндексированные пары.
2. Создать эмбеддинг (вопрос + ответ) через локальный embedding provider.
3. Upsert в Qdrant коллекцию `gk_qa_pairs_v1`.

## Автоответчик

### Эвристика определения вопросов

Сообщение считается вопросом, если:
- Содержит `?`
- Начинается с вопросительного слова (как, почему, где, ...)
- Содержит паттерн вроде «не работает», «что делать», «подскажите»

### Порог уверенности

Ответ отправляется только при `confidence >= GK_RESPONDER_CONFIDENCE_THRESHOLD` (0.7).

### Rate limiting

- Per-user: до 5 ответов за 120 секунд
- Per-group: до 30 ответов за 300 секунд

### Dry-run по умолчанию

По умолчанию автоответчик работает в режиме dry-run — формирует ответы и логирует их, но не отправляет в группу. Для реального режима используйте флаг `--live`.

### Ссылка на похожий кейс

При успешном ответе автоответчик добавляет ссылку на исходное сообщение в группе, где уже обсуждалась похожая проблема. Обычно это ссылка на сообщение с решением или на наиболее релевантный ответ в найденной цепочке.
