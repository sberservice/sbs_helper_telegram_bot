# Group Knowledge — Подсистема добычи знаний из групп

## Обзор

**Group Knowledge** — подсистема для автоматического извлечения технических знаний из переписок в Telegram-группах техподдержки. Работает как набор standalone-скриптов, не зависящих от основного бота.

### Что делает

1. **Коллектор** — слушает сообщения в настроенных Telegram-группах через Telethon, сохраняет текст/подписи/изображения в БД.
2. **Обработчик изображений** — описывает скриншоты через GigaChat Vision API, привязывая текстовое описание к сообщениям.
3. **Анализатор** — извлекает пары вопрос-ответ из собранных сообщений: по reply-to связям и полным цепочкам обсуждения (thread-based), а также через LLM-анализ контекста (LLM-inferred).
4. **Автоответчик** — в daemon-режиме встроен в коллектор: обнаруживает вопросы в группах и отвечает из накопленной базы Q&A-пар через RAG (с dry-run режимом по умолчанию).
	При ответе также прикладывает ссылку на похожее обсуждение в группе, из которого была найдена релевантная Q&A-пара.

### Архитектура

```
scripts/
├── gk_collector.py        ← Standalone: сбор сообщений + обработка изображений + встроенный автоответчик daemon
├── gk_analyze.py          ← CLI: извлечение Q&A-пар + индексация в Qdrant
└── gk_responder.py        ← Standalone: legacy-автоответчик для отдельного запуска

src/group_knowledge/
├── __init__.py
├── settings.py            ← Константы и метаданные модуля
├── models.py              ← Модели данных (GroupMessage, QAPair, ...)
├── database.py            ← Слой работы с MySQL
├── collector_responder.py ← Склейка соседних сообщений и мост к автоответчику
├── image_processor.py     ← Скачивание + описание изображений (GigaChat)
├── message_collector.py   ← Telethon listener + управление группами
├── qa_analyzer.py         ← Извлечение Q&A пар (thread + LLM)
├── qa_search.py           ← Гибридный поиск по Q&A базе

└── responder.py           ← Автоответчик с rate limiting
```

## Быстрый старт

### 1. Настройка окружения

Добавьте в `.env`:

```env
# GigaChat для описания изображений
# Здесь нужен Authorization key / Basic key из кабинета Sber Developers,
# а не access token и не обычный API key.
GIGACHAT_CREDENTIALS=your_gigachat_authorization_key
GIGACHAT_SCOPE=GIGACHAT_API_PERS  # или GIGACHAT_API_CORP

# Telethon (если ещё не настроен)
TELETHON_API_ID=12345
TELETHON_API_HASH=abc123def456
```

### 2. Создание таблиц в БД

```bash
mysql -u root -p archie_db < sql/group_knowledge_setup.sql

# (для существующих установок) добавить поля классификации вопроса в gk_messages
mysql -u root -p archie_db < sql/group_knowledge_question_classification_setup.sql

# (опционально) очистка старых дублей и защита от новых
mysql -u root -p archie_db < sql/group_knowledge_qa_question_unique.sql
```

### 3. Настройка групп

```bash
python scripts/gk_collector.py --manage-groups
```

Интерактивное CLI для добавления/удаления групп. Группы можно выбрать прямо из списка
доступных групп и супергрупп аккаунта Telegram. Конфигурация сохраняется в `config/gk_groups.json`.

> Важно: при первой авторизации Telethon код подтверждения обычно приходит в приложение Telegram
> (служебный чат `Telegram`), а не по SMS. Номер телефона вводите в международном формате,
> например `+79991234567`.

> Важно для GigaChat: в `GIGACHAT_CREDENTIALS` нужно указывать именно `Authorization key`
> (Basic key) из кабинета Sber Developers. Если указать access token или неподходящий ключ,
> GigaChat вернёт ошибку `Authorization error: header is incorrect`.

### 4. Запуск коллектора

```bash
# Слушатель новых сообщений + встроенный автоответчик в dry-run режиме
python scripts/gk_collector.py

# Реально отправлять ответы из встроенного автоответчика
python scripts/gk_collector.py --live

# Только сборщик без встроенного автоответчика
python scripts/gk_collector.py --collect-only

# Test mode: слушать test group, но использовать базу знаний real group
python scripts/gk_collector.py --test-mode

# Или загрузка истории за 7 дней
python scripts/gk_collector.py --backfill --days 7

# Принудительно пересобрать уже загруженные сообщения,
# заново скачать изображения и пересоздать их описания
python scripts/gk_collector.py --backfill --days 7 --force
```

Флаг `--force` полезен, если нужно повторно прогнать GigaChat по изображениям,
перекачать медиафайлы или обновить уже собранные сообщения без ручной очистки БД.

В daemon-режиме `gk_collector.py` теперь выполняет две задачи одновременно:
- сохраняет новые сообщения и медиа в БД;
- запускает встроенный автоответчик, если не указан `--collect-only`.

Перед стартом live-listener коллектор также автоматически добирает сообщения,
которые успели появиться в группе с прошлого запуска. Для этого он берёт из БД
последний сохранённый `telegram_message_id` по каждой группе и догружает все более
новые сообщения. Если группа ещё ни разу не собиралась, автоматический добор не
выполняется — для первичного наполнения используйте `--backfill`.

В логах collector для каждого сохранённого сообщения теперь дополнительно показывается
исходный timestamp самого сообщения, чтобы было видно фактическое время его отправки,
а не только время записи в БД.

Для обычных сообщений daemon ждёт короткое окно склейки и объединяет соседние сообщения
одного и того же пользователя в один запрос. Это позволяет корректно обработать сценарии,
когда пользователь сначала отправляет изображение, а через несколько секунд — текст,
или наоборот. Команда `/qa ...` обрабатывается сразу, без ожидания окна склейки.

### 5. Анализ и извлечение Q&A

```bash
# Анализ за вчера (по умолчанию)
python scripts/gk_analyze.py

# Конкретная дата
python scripts/gk_analyze.py --date 2024-01-15

# Диапазон дат
python scripts/gk_analyze.py --date-range 2024-01-10 2024-01-15

# Все даты с необработанными сообщениями (processed = 0)
python scripts/gk_analyze.py --all-unprocessed

# Только индексация в Qdrant
python scripts/gk_analyze.py --index

# Принудительный переанализ (включая уже processed сообщения)
python scripts/gk_analyze.py --date 2024-01-15 --force-reanalyze
```

`gk_analyze.py` по умолчанию анализирует только сообщения с `processed = 0`.
Это делает повторные запуски идемпотентными: уже обработанные сообщения не будут
повторно извлекаться в новые Q&A-пары.
Если нужен повторный разбор исторических данных, используйте флаг `--force-reanalyze`.
Для прохода по всем накопленным необработанным сообщениям без ручного указания даты используйте `--all-unprocessed`.

Новые сообщения при сборе теперь дополнительно классифицируются как вопрос/не вопрос.
Для сообщений без символа `?` коллектор отправляет текст в LLM, сохраняет результат в `gk_messages`
(`is_question`, `question_confidence`, `question_reason`, `question_model_used`, `question_detected_at`),
а анализатор позже использует эти метки как подсказки `QUESTION_HINT` / `NOT_QUESTION_HINT` во время LLM-inferred анализа.
Сообщения от sender-id из `GK_IGNORED_SENDER_IDS` анализатор исключает до построения цепочек,
чтобы служебные сообщения бота не попадали в Q&A-пары.

### 6. Запуск автоответчика

```bash
# Основной рекомендованный режим: встроенный автоответчик внутри коллектора
python scripts/gk_collector.py

# Live режим — реально отправляет ответы
python scripts/gk_collector.py --live

# Test mode — выбрать test group и real group через интерактивное меню
python scripts/gk_collector.py --test-mode
```

Встроенный автоответчик в `gk_collector.py` работает так:
- команда `/qa ...` всегда обрабатывается сразу и считается вопросом принудительно;
- обычные сообщения сначала буферизуются по паре `(group_id, sender_id)` на короткое окно склейки;
- текст и изображения одного пользователя объединяются в один запрос;
- если явного `?` нет, используется сохранённая или онлайн LLM-классификация вопроса.

Пример запроса через команду:

```text
/qa как исправить ошибку 1001 на терминале?
```

Сообщения без префикса `/qa` не игнорируются полностью: daemon collector может ответить и на них,
если после склейки и классификации сообщение распознано как технический вопрос.

В `--test-mode` встроенный автоответчик работает как daemon: после интерактивного выбора связки
`test group -> real group` он постоянно слушает новые сообщения в тестовой группе,
но логически считает их сообщениями реальной группы.

Скрипт `gk_responder.py` сохранён для отдельного legacy-запуска, но основной эксплуатационный
сценарий теперь проходит через daemon `gk_collector.py`.

### 7. Очистка данных по группе

```bash
# Предпросмотр (dry-run): показать, сколько записей будет удалено
python scripts/gk_delete_group_data.py --group-id -1001234567890

# Интерактивное меню выбора группы (если --group-id не передан, режим включается автоматически)
python scripts/gk_delete_group_data.py --interactive

# Реальное удаление данных группы
python scripts/gk_delete_group_data.py --group-id -1001234567890 --yes

# Реальное удаление без очистки векторных точек
python scripts/gk_delete_group_data.py --group-id -1001234567890 --yes --no-vector-cleanup
```

Утилита удаляет данные группы из `gk_messages`, `gk_qa_pairs`, `gk_responder_log`,
связанные записи `gk_image_queue`, а также по умолчанию очищает точки Q&A в коллекции
`gk_qa_pairs_v1` в Qdrant.
В интерактивном режиме дополнительно запрашивается подтверждение удаления через ввод `yes`.

## Конфигурация

### Основные настройки (`config/ai_settings.py`)

| Параметр | Умолчание | Описание |
|----------|-----------|----------|
| `GIGACHAT_CREDENTIALS` | `.env` | API-ключ GigaChat |
| `GIGACHAT_MODEL` | `GigaChat-Pro` | Модель GigaChat для описания изображений |
| `GK_IMAGE_STORAGE_PATH` | `./data/group_knowledge/images` | Путь хранения скачанных изображений |
| `GK_DRY_RUN` | `True` | Глобальный dry-run для автоответчика |
| `GK_RESPONDER_CONFIDENCE_THRESHOLD` | `0.7` | Минимальная уверенность для ответа |
| `GK_QA_VECTOR_COLLECTION` | `gk_qa_pairs_v1` | Коллекция Qdrant для Q&A пар |
| `GK_RESPONDER_TOP_K` | `5` | Число Q&A пар для генерации ответа |
| `GK_ANALYSIS_BATCH_SIZE` | `50` | Размер батча для LLM-анализа |
| `GK_IGNORED_SENDER_IDS` | `111111` | Список sender-id через запятую, которые нужно исключать из анализа и автоответов |
| `GK_MESSAGE_GROUPING_WINDOW_SECONDS` | `6` | Окно склейки соседних сообщений одного пользователя в daemon collector |

### Rate limiting (`src/common/constants/sync.py`)

| Параметр | Умолчание | Описание |
|----------|-----------|----------|
| `GK_RATE_LIMIT_USER_MAX` | `5` | Макс. ответов на пользователя в окне |
| `GK_RATE_LIMIT_USER_WINDOW` | `120` с | Окно rate limit для пользователя |
| `GK_RATE_LIMIT_GROUP_MAX` | `30` | Макс. ответов на группу в окне |
| `GK_RATE_LIMIT_GROUP_WINDOW` | `300` с | Окно rate limit для группы |

### Telethon-сессии

Коллектор и standalone-автоответчик используют отдельные Telethon-сессии:
- `gk_collector_session` — для коллектора
- `gk_responder_session` — для legacy-скрипта автоответчика

Сессия `helper_session` намеренно не переиспользуется: это снижает риск sqlite-конфликтов Telethon вида `database is locked`, если параллельно работает другой скрипт.

Если предыдущий запуск был остановлен через `Ctrl+C`, следующий старт всё ещё может упереться в временный lock session-файла. В этом случае Group Knowledge завершится с понятной ошибкой, и запуск нужно повторить вручную после освобождения файла.

При первом запуске потребуется авторизация по номеру телефона.
Код подтверждения обычно приходит в приложение Telegram, а если на аккаунте включена
двухэтапная аутентификация, после кода потребуется ввести облачный пароль.

## Структура БД

### gk\_messages

Собранные сообщения из групп.

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | BIGINT | Внутренний ID |
| `telegram_message_id` | BIGINT | ID сообщения в Telegram |
| `group_id` | BIGINT | ID группы |
| `sender_id` | BIGINT | ID отправителя |
| `message_text` | TEXT | Текст сообщения |
| `caption` | TEXT | Подпись к медиа |
| `has_image` | TINYINT | Есть ли изображение |
| `image_description` | TEXT | Описание изображения от GigaChat |
| `reply_to_message_id` | BIGINT | ID сообщения-ответа |
| `processed` | TINYINT | Обработано анализатором |
| `is_question` | TINYINT | LLM/правило определило сообщение как вопрос |
| `question_confidence` | FLOAT | Уверенность классификатора вопроса |
| `question_reason` | VARCHAR(512) | Краткая причина классификации |
| `question_model_used` | VARCHAR(128) | Модель/правило классификации |
| `question_detected_at` | BIGINT | Время классификации вопроса |

### gk\_qa\_pairs

Извлечённые пары вопрос-ответ.

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | BIGINT | Внутренний ID |
| `question_text` | TEXT | Текст вопроса |
| `answer_text` | TEXT | Текст ответа |
| `extraction_type` | ENUM | `thread_reply` / `llm_inferred` |
| `confidence` | FLOAT | Уверенность (0.0–1.0) |
| `approved` | TINYINT | Одобрена (по умолчанию 1) |
| `indexed` | TINYINT | Проиндексирована в Qdrant |

Для предотвращения дублей по одному и тому же исходному вопросу рекомендуется
применить миграцию `sql/group_knowledge_qa_question_unique.sql`.
Она удаляет старые дубли по `question_message_id` (оставляя самую новую запись)
и добавляет `UNIQUE`-ограничение на `question_message_id`.

### gk\_image\_queue

Очередь обработки изображений.

### gk\_responder\_log

Лог ответов автоответчика (включая dry-run).

## Алгоритм извлечения Q&A

### Thread-based (Phase 1)

1. Найти все reply-цепочки за день и собрать полные обсуждения от корневого вопроса до финального решения.
2. Включить в цепочку уточняющие вопросы, промежуточные ответы и служебные reply-сообщения.
3. Дополнительно подтянуть соседние (nearby) последовательные сообщения от тех же участников цепочки, даже если между ними нет Telegram reply-связи.
4. Если после nearby-расширения несколько кандидатов сильно пересекаются (overlap не менее 60% относительно меньшей цепочки), объединить их в одну цепочку, чтобы собрать ответы всех участников в общий контекст; root у объединённой цепочки выбирается более ранний.
5. Передать в LLM не только текст сообщений, но и время каждого сообщения, а также имя автора, чтобы модели было проще понять ход обсуждения.
6. Отделить итоговое решение от благодарностей и прочего шума.
7. При сохранении вопроса в БД дополнительно включить gist проблемы с изображения, если исходный вопрос сопровождался скриншотом или фото с описанием от GigaChat.
8. Валидировать всю цепочку через LLM: является ли она полезным техническим Q&A?
9. Если валидна — сохранить одну итоговую очищенную пару вопрос-ответ по всей цепочке.

### LLM-inferred (Phase 2)

1. Собрать сообщения, не вошедшие в thread-пары.
2. Разбить на батчи по `GK_ANALYSIS_BATCH_SIZE`.
3. Отправить батч в LLM с промптом поиска семантических Q&A пар.
4. Сохранить найденные пары.

### Индексация (Phase 3)

1. Найти новые неиндексированные пары.
2. Создать эмбеддинг (вопрос + ответ) через локальный embedding provider.
3. Сохранить каждую пару как отдельный векторный документ в Qdrant коллекцию `gk_qa_pairs_v1`.
4. При поиске по этой коллекции затем подтянуть полную Q&A-пару из MySQL по `pair_id`, чтобы автоответчик работал на актуальных данных из БД.

## Автоответчик

### Эвристика определения вопросов

Сообщение считается вопросом, если:
- Содержит `?`
- Начинается с вопросительного слова (как, почему, где, ...)
- Содержит паттерн вроде «не работает», «что делать», «подскажите»
- Или было определено общей LLM-классификацией как технический вопрос без явного `?`

### Склейка соседних сообщений

В daemon collector соседние сообщения одного пользователя в одной группе буферизуются на
`GK_MESSAGE_GROUPING_WINDOW_SECONDS` секунд и затем объединяются в один запрос. Если в окне
есть сообщение только с изображением без текста, в объединённый вопрос добавляется маркер
`[Пользователь приложил изображение без подписи]`.

### Порог уверенности

Ответ отправляется только при `confidence >= GK_RESPONDER_CONFIDENCE_THRESHOLD` (0.7).

### Rate limiting

- Per-user: до 5 ответов за 120 секунд
- Per-group: до 30 ответов за 300 секунд

### Dry-run по умолчанию

По умолчанию встроенный автоответчик работает в режиме dry-run — формирует ответы и логирует их, но не отправляет в группу. Для реального режима используйте флаг `--live`.

### Ссылка на похожий кейс

При успешном ответе автоответчик добавляет ссылку на исходное сообщение в группе, где уже обсуждалась похожая проблема. Обычно это ссылка на сообщение с решением или на наиболее релевантный ответ в найденной цепочке.

## Тестирование

```bash
pytest tests/test_group_knowledge.py tests/test_gk_responder_cli.py tests/test_gk_collector_cli.py -v
```

Тесты покрывают: модели данных, эвристику вопросов, rate limiter, парсинг JSON из LLM, валидацию Q&A пар, генерацию ответов, конфигурацию групп, CLI и bridge-логику daemon collector.
