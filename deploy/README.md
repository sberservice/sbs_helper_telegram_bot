# Развёртывание SBS Archie на Windows

Руководство по установке и эксплуатации SBS Archie на Windows-сервере с автозапуском через Task Scheduler.

## Требования

| Компонент | Версия | Назначение |
|-----------|--------|------------|
| Windows | 10/11 или Server 2019+ | Операционная система |
| Python | 3.10+ | Основная платформа |
| Git | Любая | Обновление из репозитория |
| Node.js | 18+ | Сборка frontend админ-панели |
| MySQL | 8.0+ | База данных |
| curl | Встроен в Windows 10+ | Health check watchdog |

## Структура файлов

```
deploy/
├── launch_config.json         ← Конфигурация автозапуска процессов
├── check_telethon_sessions.py ← Анализ required Telethon session-файлов
├── check_sessions.bat         ← Удобный запуск проверки сессий
├── start.bat                  ← Запуск admin_web + watchdog
├── stop.bat                   ← Остановка всех процессов
├── update.bat                 ← Обновление из GitHub
├── setup.bat                  ← Первоначальная настройка
├── setup_cuda.bat             ← Настройка GPU/CUDA-окружения для PyTorch
├── task_scheduler_setup.bat   ← Настройка Task Scheduler
├── backups/                   ← Автобэкапы конфигурации (создаётся автоматически)
└── README.md                  ← Этот файл  
```

## Первоначальная установка

### 1. Клонирование репозитория

```cmd
git clone <repository-url> C:\SBS_Archie
cd C:\SBS_Archie
```

### 2. Запуск setup.bat

```cmd
deploy\setup.bat
```

Скрипт автоматически:
- Проверит наличие Python, Git, Node.js
- Создаст виртуальное окружение (`venv_name`)
- Установит Python-зависимости
- Соберёт frontend админ-панели
- Создаст необходимые папки (`logs`, `data`)
- Скопирует `.env.example` → `.env`

### 2.1 Настройка CUDA-окружения (для серверов с NVIDIA GPU)

Если планируется использовать GPU для эмбеддингов/векторного поиска, после `setup.bat`
выполните отдельный скрипт CUDA-настройки:

```cmd
deploy\setup_cuda.bat
```

Скрипт:
- Проверяет доступность `nvidia-smi`
- Создаёт/переиспользует `venv_name`
- Устанавливает зависимости без `torch`
- Устанавливает CUDA-сборку `torch/torchvision/torchaudio` из индекса PyTorch (`cu124`)
- Проверяет `torch.cuda.is_available() == True`

Важно: для CUDA-сборки рекомендуется Python `3.11` или `3.12`.
Если в `venv_name` обнаружен Python `3.13+`, `setup_cuda.bat` автоматически пересоздаст venv на `3.12` (если установлен `py -3.12`).

### 3. Настройка конфигурации

Отредактируйте `config\.env` — укажите:
- `TELEGRAM_BOT_TOKEN` — токен бота
- `DB_HOST`, `DB_USER`, `DB_PASSWORD`, `DB_NAME` — подключение к MySQL
- `AI_DEEPSEEK_API_KEY` — ключ API DeepSeek
- Остальные настройки по необходимости

### 3.1 Проверка/подготовка Telethon-сессий

Перед первым рабочим запуском выполните:

```cmd
deploy\check_sessions.bat
```

Если каких-то session-файлов не хватает, скрипт покажет точные команды для их создания (например, `python scripts/gk_collector.py --manage-groups`, `python scripts/the_helper.py --manage-groups`).
Также `deploy\check_sessions.bat` умеет запустить эти команды автоматически через интерактивное меню.
Проверка учитывает не только наличие файла `*.session`, но и факт авторизации Telegram для этой сессии.

Важно: session-файлы можно перенести с предыдущего сервера (`*.session` в корень проекта), тогда повторная авторизация не потребуется.

Важно: daemon-процессы (`gk_collector`, `the_helper`) **не** выполняют интерактивную авторизацию Telethon при автозапуске. Если сессия отсутствует, запуск будет остановлен с подсказкой, какую команду выполнить вручную.

### 4. Настройка базы данных

```cmd
mysql -u root -p < schema.sql
```

Затем выполните необходимые скрипты из папки `sql\`.

### 5. Настройка launch_config.json

Файл `deploy\launch_config.json` определяет, какие процессы запускаются автоматически при старте:

```json
{
    "processes": {
        "telegram_bot": { "enabled": true, "preset": "Стандартный запуск" },
        "gk_collector": { "enabled": true, "preset": "Dry-run" },
        "the_helper":   { "enabled": false, "preset": null }
    }
}
```

Редактировать можно:
- Вручную в текстовом редакторе
- Через админ-панель: `http://localhost:8090` → Менеджер процессов → вкладка «Запуск»

### 6. Проверка работы

```cmd
deploy\start.bat
```

`start.bat` автоматически проверяет обязательные Telethon-сессии и останавливает запуск при их отсутствии с подсказкой, как их создать.
При автозапуске используется non-interactive режим проверки (`deploy\check_sessions.bat --non-interactive`), чтобы сервер не зависал в диалоге ввода.

Если React build отсутствует (`admin_web/frontend/dist/index.html`), `start.bat` автоматически попытается выполнить `npm install` и `npm run build`, чтобы UI был доступен.

Откройте `http://localhost:8090` для локального доступа к админ-панели.

Для доступа из LAN используйте IP адрес сервера:
- `http://<IP_СЕРВЕРА>:8090`

Важно: откройте входящий TCP-порт `8090` в Windows Firewall.

### 7. Настройка автозапуска

Запустите **от имени администратора**:

```cmd
deploy\task_scheduler_setup.bat
```

Создаст задачу `SBS_Archie` в Task Scheduler:
- Триггер: при загрузке системы
- Задержка: 2 минуты (ожидание MySQL и сети)
- Права: наивысшие

## Ежедневная эксплуатация

### Автоматический запуск

Компьютер перезагружается (например, в 7:00) → через 2 минуты Task Scheduler запускает `start.bat` → admin_web стартует и запускает процессы из `launch_config.json`.

При запуске открывается отдельное окно `SBS_Archie_AdminWeb` с live-выводом сервера.

### Архитектура запуска

```
Task Scheduler
  └─ start.bat (watchdog)
       └─ admin_web (оркестратор, порт 8090)
            ├─ telegram_bot
            ├─ image_queue
            ├─ soos_queue
            ├─ health_check_daemon
            ├─ gk_collector
            └─ the_helper
```

- **admin_web** — единая точка управления всеми процессами
- **start.bat** — watchdog: следит за admin_web через health endpoint, перезапускает при падении (до 5 попыток)
- **launch_config.json** — декларативная конфигурация: какие процессы запускать и с какими параметрами

### Ручное управление

| Действие | Команда |
|----------|---------|
| Запуск | `deploy\start.bat` |
| Остановка | `deploy\stop.bat` |
| Обновление | `deploy\update.bat` |
| Статус | Открыть `http://localhost:8090` → Менеджер процессов |

### Управление через админ-панель

Админ-панель (`http://localhost:8090` локально или `http://<IP_СЕРВЕРА>:8090` из LAN) позволяет:
- Запускать/останавливать отдельные процессы
- Менять пресеты и флаги запуска
- Редактировать конфигурацию автозапуска (вкладка «Запуск»)
- Просматривать логи процессов

## Обновление

```cmd
deploy\update.bat
```

Скрипт автоматически:
1. Остановит все процессы через API
2. Создаст бэкап `.env`, `launch_config.json`, файлов групп
3. Выполнит принудительную синхронизацию с `origin/main` (`git reset --hard origin/main`, локальные tracked-изменения будут отброшены)
4. Обновит Python-зависимости (базовые пакеты + отдельная установка `torch`-стека без двойной переустановки)
5. Пересоберёт frontend
6. Предложит запустить бота

Бэкапы хранятся в `deploy\backups\` (автоочистка через 30 дней).

На GPU-серверах `deploy\update.bat` автоматически ставит CUDA-сборку `torch` (`cu124`),
а на CPU-серверах — CPU-сборку. Перед установкой проверяется текущий стек,
и если версии уже совпадают, переустановка пропускается.

`deploy\setup_cuda.bat` нужен для первичной настройки CUDA или ручного восстановления,
если автоматическая проверка в `update.bat` завершилась предупреждением.

## Логи

| Лог | Путь |
|-----|------|
| admin_web (исторический/предыдущие запуски) | `logs\admin_web.log` |
| Логи процессов | Через admin_web → Менеджер процессов → Логи |

При каждом запуске `start.bat` старый лог admin_web ротируется с добавлением даты. Логи старше 7 дней удаляются автоматически.

## Устранение неполадок

### admin_web не запускается

1. Проверьте лог: `logs\admin_web.log`
2. Убедитесь, что порт 8090 не занят: `netstat -an | findstr 8090`
3. Проверьте `.env`: подключение к БД, токены
4. Запустите вручную для диагностики:
   ```cmd
   cd C:\SBS_Archie
   venv_name\Scripts\activate
   python -m admin_web
   ```

### Процессы не запускаются после обновления

1. Проверьте `deploy\launch_config.json` — файл мог быть изменён
2. Проверьте ошибки в админ-панели → Менеджер процессов
3. Убедитесь, что зависимости установлены: `pip install -r requirements.txt`

### Telethon-сессии

Имена файлов `*.session` берутся из env (например, `GK_COLLECTOR_SESSION_NAME`, `HELPER_SESSION_NAME`, `TELETHON_SESSION_NAME`; по умолчанию — `gk_collector_session`, `helper_session`, `chat_sync_session`).
Сами `*.session` файлы привязаны к конкретной машине. При переносе на новый компьютер:
1. Скопируйте файлы сессий на новую машину
2. Или пересоздайте сессии (потребуется SMS-верификация)

Проверка перед запуском:
```cmd
deploy\check_sessions.bat
```

Если код Telegram не приходит при создании сессии, запустите диагностическую утилиту:
```cmd
python scripts\telethon_login_debug.py --phone +79991234567
```
Утилита пошагово проверяет отправку кода, показывает типичные причины \
(`FLOOD_WAIT`, неверный формат телефона, неверный код, 2FA) и даёт явные подсказки.
По умолчанию имя session берётся из `GK_COLLECTOR_SESSION_NAME` в env; при необходимости можно переопределить через `--session-name`.

### git pull конфликты

`update.bat` не использует `git pull` и автоматически сбрасывает локальные tracked-изменения через `git reset --hard origin/main`.

Если нужно сохранить локальные правки перед обновлением — сделайте это вручную до запуска скрипта:
```cmd
cd C:\SBS_Archie
git add .
git commit -m "local changes backup"
```

## Задача Task Scheduler

### Проверка

```cmd
schtasks /Query /TN "SBS_Archie" /V
```

### Ручной запуск задачи

```cmd
schtasks /Run /TN "SBS_Archie"
```

### Удаление задачи

```cmd
schtasks /Delete /TN "SBS_Archie" /F
```

### Ручное создание (если скрипт не работает)

1. Откройте `taskschd.msc`
2. «Создать задачу» (не простую)
3. Вкладка «Общие»: имя `SBS_Archie`, «Выполнять с наивысшими правами»
4. Вкладка «Триггеры»: «При запуске» → задержка 2 минуты
5. Вкладка «Действия»: запуск `C:\SBS_Archie\deploy\start.bat`
6. Вкладка «Параметры»: разрешить запуск по требованию
